<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| local filesystem → git index | `git rm --cached` changes what git tracks without touching files on disk; a mistake here could either leave sensitive/IDE-local state tracked or (if `--cached` were omitted) delete on-disk artifacts a developer still needs locally. |
| pre-move history → post-move history | Directory restructuring is the one operation in this task that can silently and irreversibly sever `git blame`/`git log` continuity if done as delete+recreate instead of a tracked rename. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-quick-01 | Repudiation | `git mv` directory restructure (Task 2) | medium | mitigate | Use `git mv` (never manual delete+recreate) for all three project directories so git's rename detection preserves commit history end-to-end; verified explicitly via `git log --follow` in Task 3 rather than assumed. |
| T-quick-02 | Information Disclosure | Tracked `.idea/` and `obj/` artifacts (Task 1) | low | mitigate | `git rm --cached` (not `git rm`) removes IDE/build-artifact files from version control while leaving them on local disk untouched; `.idea/` is added to `.gitignore` so IDE workspace state and machine-local paths are never re-tracked going forward. |
| T-quick-03 | Tampering | `SentinelSuite.slnx` / `.csproj` `ProjectReference` paths (Task 2) | medium | mitigate | Task 3 runs `dotnet build` and `dotnet test` immediately after the path edits, failing fast on any broken reference before the change is ever committed, rather than trusting the edits by inspection alone. |

No package-manager installs occur in this task (no npm/pip/cargo), so the package legitimacy gate does not apply.
</threat_model>
