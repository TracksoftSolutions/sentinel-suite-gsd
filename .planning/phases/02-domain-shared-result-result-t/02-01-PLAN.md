**Naming-collision note (resolve exactly this way, do not deviate):** D-04 requires an instance property literally named `.Error` (first-error convenience accessor, `result.Error.Message`), and D-09 lists a named static factory `Result.Error(...)` for the generic `ResultStatus.Error` status. A property and a static method cannot share the same identifier on the same C# type — the C# compiler rejects it with CS0102 ("The type 'Result' already contains a definition for 'Error'"), confirmed by a direct local compile during planning. Resolve this by naming the generic-failure static factory `Result.Failure(params Error[] errors)` instead of `Result.Error(...)`, while keeping the `.Error` instance property's name exactly as D-04 specifies. `Failure` is not an invented name: it is CSharpFunctionalExtensions' own actual name for this exact "generic, non-further-categorized failure" factory (one of this phase's two co-equal reference libraries per RESEARCH.md), so this is a grounded substitution, not an arbitrary one. `Failure` still satisfies D-09's core intent — one dedicated named factory per status, not a single generic `Fail(status, errors)` dispatcher. Document this rationale inline in `Result.cs`'s XML remarks so a future reviewer (or the 02-02 plan building `Result<T>`, which must mirror this exact naming) does not "fix" it back to `Error` and reintroduce the compile error.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Future Result → external response mapping (no boundary exists yet this phase) | `Result.Exception` (populated only by `CriticalError`) is a latent landmine for whichever future phase adds an API/Application layer that serializes a `Result` into an HTTP response — no live trust boundary exists in `Domain.Shared` itself yet |
| Any future caller catching an exception and converting it to `Result.CriticalError(...)` | The caught exception's raw `.Message` is a potential source of internal-state leakage (file paths, connection details, stack-adjacent text) if echoed unfiltered into a business-facing `Error.Message` |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|------------|----------|--------------|------------------|
| T-2-01 | Information Disclosure | `Result.Exception` property (populated only by the `CriticalError` factory) | medium | mitigate | `Exception` is a distinct top-level property, never folded into `Errors`/`Error`, so it can be explicitly stripped at a future serialization boundary; XML remarks on `Result`/`CriticalError` document that `Exception` must never appear in any external-facing serialization of a `Result`, flagged explicitly for the future API-layer phase's own security research. `ResultCriticalErrorTests` confirms the property is populated only by `CriticalError` and left `null` by every other factory, so its presence is always intentional and detectable. |
| T-2-02 | Information Disclosure | `CriticalError`'s default `Error.Message` fallback (RESEARCH.md Pitfall 5) | medium | mitigate | `CriticalError` never passes a null/empty `exception.Message` straight into the required-non-empty `Error.Message` field (D-03); it falls back to a fixed literal ("An unexpected error occurred.") instead, preventing both a D-03 violation and an uncontrolled leak of whatever empty/default `Message` an internal exception type happens to return. Verified by `ResultCriticalErrorTests`' empty-message-exception-fixture case. |
| T-2-03 | Tampering | `Result.cs`, `Error.cs`, `ResultStatus.cs` (pure data/factory types, no runtime external input this phase) | low | accept | These are pure construction/inspection types with no I/O and no untrusted input surface at this layer; real business-rule input validation happens in whatever future Domain/UseCases code calls these factories, not in `Result` itself. |

</threat_model>
