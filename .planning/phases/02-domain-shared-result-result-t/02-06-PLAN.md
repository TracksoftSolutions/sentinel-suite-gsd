<objective>
Add `Result.Combine(params Result[] results)` and `Result.Combine<T>(params Result<T>[] results)` — the all-or-nothing batch aggregator D-15 requires, distinct from the short-circuit-on-first-failure chaining the other Wave 3 plans (02-03 Map/Bind, 02-04 Ensure/Match, 02-05 OnSuccess/OnFailure) provide. `Combine` succeeds only when every input succeeds; on any failure it unions **every** failed input's `Errors` into one failed `Result`, matching D-15's "validate everything, report every problem at once" batch semantics and the list-of-structured-errors shape already established by D-01.

**Overload strategy (RESEARCH.md Pitfall 4, locked scope boundary):** Two separate overloads — `Combine(params Result[])` and `Combine<T>(params Result<T>[])` — not Ardalis.Result's `Result : Result<Result>` self-referential-inheritance trick. That inheritance shape is incompatible with D-16 (both `Result` and `Result<T>` must be sealed; a sealed type cannot be a non-sealed base of anything). Both overloads project each input's `.Errors`, flatten, and construct one aggregate failed `Result` — no inheritance relationship between `Result` and `Result<T>` is introduced anywhere in this plan.

**Placement decision (Claude's discretion, per CONTEXT.md — document here so it is not "corrected" later):** D-15's own phrasing — and this phase's plan outline — names the target API literally as `Result.Combine(...)`. A true C# extension method cannot provide that call syntax for a `params`-array-of-the-extended-type signature (extension methods require an instance `this` receiver; `Combine`'s array-of-many-instances shape has no single receiver to extend). RESEARCH.md's Recommended Project Structure lists a `ResultCombineExtensions.cs` file as directional guidance for where this logic conceptually belongs, but the only way to make `Result.Combine(...)` and `Result.Combine<T>(...)` both compile and read exactly as written is to add them as two more `public static` methods directly on the existing sealed `Result` class in `Result.cs` — the same class that already exposes `Success`, `Failure`, `Invalid`, etc. as static factories (plan 02-01). This plan therefore modifies `Result.cs` in place rather than creating a new extensions file; do not revert this to a separate `ResultCombineExtensions` static class under the assumption that RESEARCH.md's file list is prescriptive about *placement* — it documents the *pattern*, not the exact file, and CONTEXT.md's Claude's Discretion note explicitly leaves Combine's internal implementation details to planning.

Purpose: Complete this phase's combinator surface (D-12 through D-15) with the one aggregator that is fundamentally different in shape from Map/Bind/Ensure/Match/OnSuccess/OnFailure — a batch operation over N independent inputs rather than a chain over one.
Output: `Result.cs` (from plan 02-01) extended with `Combine`/`Combine<T>`; `ResultCombineTests.cs` (new) green in the mirrored test project.
</objective>
