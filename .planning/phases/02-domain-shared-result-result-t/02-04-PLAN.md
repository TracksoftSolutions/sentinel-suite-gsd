<objective>
Build the `Ensure` and `Match` railway combinators (D-12) on top of `Result`/`Result<T>` from plans 02-01/02-02, each with the full sync + Left-async + Right-async + Both-async overload set (D-13), following RESEARCH.md's Pattern 2 Left/Right/Both async-overload-split convention (the same one used for `Bind`/`Map` in plan 02-03). This plan covers ONLY `Ensure` and `Match` — `Map`/`Bind` (02-03), `OnSuccess`/`OnFailure` (02-05), and `Combine` (02-06) are separate, parallel Wave 3 plans with zero file overlap.

**Ensure's failure status (Claude's Discretion, documented here so it is not re-litigated):** D-12 illustrates `Ensure` with `result.Ensure(x => x.IsValid, error)`, framed in CONTEXT.md as pairing "naturally with Map/Bind for validation-in-a-chain." Neither CONTEXT.md nor RESEARCH.md locks which `ResultStatus` a failed predicate should produce. This plan uses `ResultStatus.Invalid` (via `Result.Invalid(error)` / `Result<T>.Invalid(error)`) for every `Ensure` failure — the natural fit for "a validation predicate rejected an otherwise-successful value," consistent with D-08's status vocabulary and CONTEXT.md's own validation framing. Do not use `Failure`/`Error` status for `Ensure`'s failure branch.

**Ensure applies to both `Result` and `Result<T>`:** D-12's example uses a value-taking predicate (`Result<T>` shape), but this plan also implements `Ensure` on the non-generic `Result` using a value-less `Func<bool>` predicate (checks an external condition mid-chain, since `Result` carries no value to test). Both variants are required so the combinator surface is complete and symmetric with `Result`/`Result<T>`'s existing factory parity (D-10).

**Match's async overload narrowing (RESEARCH.md Open Question 1 — resolved explicitly here, not left ambiguous):** Unlike `Bind`/`Ensure` (which return `Result`/`Result<T>` and can mix a sync predicate/continuation independently of the source's sync/async-ness), `Match` collapses to a plain `TOut` via *two* handlers (`onSuccess`, `onFailure`). Allowing each handler to independently be sync or async would explode the overload count (4 source shapes x 4 handler-sync/async combinations). This plan deliberately narrows that: for the Right-async and Both-async shapes, `onSuccess` and `onFailure` become async (`Task<TOut>`-returning) **together** — never one sync and the other async on the same overload. This still yields the standard 4 shapes (sync/sync, Left-async, Right-async, Both-async) per type, just with the async-continuation shapes requiring both handlers to share the same sync/async-ness. Document this narrowing in `ResultMatchExtensions.cs`'s class-level XML remarks so a future contributor doesn't "complete" a partial-async overload set by analogy to `Bind`.

Purpose: Give every future Domain/UseCases consumer a way to gate a chain on a runtime predicate (`Ensure`) and to collapse a `Result`/`Result<T>` into a single terminal value at a chain's boundary (`Match`) — both with zero-friction sync/async chaining per D-13's front-loaded async requirement.
Output: `ResultEnsureExtensions.cs` and `ResultMatchExtensions.cs` compiling in `SentinelSuite.Framework.Domain.Shared/Results/`, with `ResultEnsureTests.cs` and `ResultMatchTests.cs` (each folding sync and async overload coverage into one file, per this phase's outline deviation from RESEARCH.md's literal file layout — keeps this plan's files non-overlapping with 02-03/02-05/02-06) green in the mirrored test project.
