<objective>
Build the `Map` and `Bind` railway-style combinators (D-12, D-13) for the `Result`/`Result<T>` family completed in plans 02-01 and 02-02: `Map` transforms a success value (or produces one on a valueless success), short-circuiting on failure; `Bind` chains to another `Result`-returning operation, flattening instead of nesting, short-circuiting on failure. Each combinator ships as two variants — a generic `Result<TIn> -> Result<TOut>` form and a non-generic `Result` form — and each variant covers all 4 sync/async shapes from RESEARCH.md's Pattern 2 (sync source/sync continuation; Left-async — `Task<Result<TIn>>`/`Task<Result>` source with a sync continuation; Right-async — sync source with a `Task`-returning continuation; Both-async — both operands async), 16 methods total across two files.

**Scope boundary (do not exceed):** This plan covers ONLY `Map` and `Bind`. `Ensure`, `Match`, `OnSuccess`/`OnFailure`, and `Combine` are separate plans (02-04, 02-05, 02-06) — do not implement any part of their surface here, and do not add a `Then` alias method: D-12 lists "Bind/Then" as one combinator with two names for the same railway mechanism; RESEARCH.md's Recommended Project Structure names the single file `ResultBindExtensions.cs` (no separate `ResultThenExtensions.cs`), so `Bind` is the sole implemented name — document this naming choice inline so `Then` is never "completed" as a second method later.

**Short-circuit fidelity note (deliberate, documented simplification — do not silently "fix"):** When the generic `Result<TIn> -> Result<TOut>` variant of `Map`/`Bind` short-circuits a failed source into a new `Result<TOut>`, only the source's `.Errors` list is propagated via the generic `Failure(...)` factory — the source's original `Status` (e.g. `NotFound`, `Conflict`, `CriticalError`) and, for a `CriticalError` source, its `Exception`, are NOT preserved through the type change (they collapse to `ResultStatus.Error`/null `Exception`). This matches RESEARCH.md's own cited Pattern 2 code example (`Result<TOut>.Error(result.Errors.ToArray())`, adapted here to `Failure`) verbatim and is the correct, minimal-surface choice for this plan — `Result<TIn>` and `Result<TOut>` are different closed generic types, so preserving status/exception through the type change would require dispatching through every named factory by a `Status` switch, which is out of this plan's scope. The non-generic `Result -> Result` `Bind` variant has no such gap: on failure it returns the original `Result` instance unchanged, so `Status`/`Exception` are always preserved there. Document both facts in each file's XML remarks.

Purpose: Give Wave 3's sibling combinator plans (02-04 Ensure/Match, 02-05 OnSuccess/OnFailure, 02-06 Combine) and every future Domain-layer consumer the two most-used railway-chaining primitives, ready to compose (`result.Map(...).Bind(...)`).
Output: `ResultMapExtensions.cs` and `ResultBindExtensions.cs` compiling in `SentinelSuite.Framework.Domain.Shared/Results/`, with `ResultMapTests.cs` and `ResultBindTests.cs` green in the mirrored test project.
</objective>
