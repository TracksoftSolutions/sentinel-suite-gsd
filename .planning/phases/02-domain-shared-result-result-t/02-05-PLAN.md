<objective>
Build the `OnSuccess`/`OnFailure` side-effect combinators (D-12) for both `Result` and `Result<T>` — the two railway combinators that fire a caller-supplied action without transforming the value or the `Result` itself, unlike `Map`/`Bind` (02-03) or `Ensure`/`Match` (02-04). Each combinator ships with the full 4-shape sync/async overload matrix required by D-13 (sync source + sync action; async source + sync action, "Left-async"; sync source + async action, "Right-async"; async source + async action, "Both-async"), following RESEARCH.md Pattern 2's Left/Right/Both convention.

**Explicit narrow-vs-full-parity decision (RESEARCH.md Open Question 1 — resolved here, not left ambiguous):** The sync/async axis (4 shapes) stays at full parity across both combinators and both types, matching D-13's literal wording. The value/error-awareness axis is deliberately narrowed to one natural shape per verb: `OnSuccess` on `Result<T>` receives the value (`Action<T>`/`Func<T, Task>`, since `Value` is always safe to read on a successful instance); `OnSuccess` on non-generic `Result` and `OnFailure` on both types take a no-argument `Action`/`Func<Task>`. This narrowing is not arbitrary — `Result<T>.Value` throws on a failed instance (D-06), so an `OnFailure` overload that tried to pass `T` to its action would be unsound by construction; `Result` (non-generic) never has a value to pass in the first place. No errors-aware `OnFailure(Action<IReadOnlyList<Error>>)` overload is added: callers who need `.Errors`/`.Error` inside the failure hook already have the source `Result`/`Result<T>` available via the variable they are chaining from (e.g. `var result = DoThing(); result.OnFailure(() => log.Error(result.Error!.Message));`), so a parameterized overload would duplicate information already in scope rather than add capability. This keeps the combinator surface at 8 methods per verb (16 total) instead of doubling again for an errors-aware variant.

Purpose: Complete D-12's side-effect-hook pair so future Domain/UseCases code can attach logging/notification behavior mid-chain (`result.Ensure(...).OnFailure(...).Map(...)`) without the hook itself altering control flow.
Output: `ResultOnSuccessOnFailureExtensions.cs` compiling in `SentinelSuite.Framework.Domain.Shared/Results/`, with `ResultOnSuccessOnFailureTests.cs` (sync + all 3 async shapes, both types) green in the mirrored test project.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Caller-supplied Action/Func&lt;Task&gt; delegates invoked by OnSuccess/OnFailure | Purely in-process side-effect hooks (e.g. logging, notification) — no new serialization or network boundary is introduced by this plan; the delegate executes with whatever trust level the calling code already has, over already-validated Result/Result&lt;T&gt; instances constructed in 02-01/02-02 |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|------------------|
| T-2-07 | Information Disclosure | OnFailure action bodies that read the captured Result's Error/Errors/Exception for logging | low | accept | This plan introduces no new field, no new logging call, and no new serialization path — OnFailure's signature intentionally carries no value/error parameter, so any inspection of Errors/Exception happens entirely through the caller's own captured Result variable and remains governed by 02-01's existing CriticalError.Exception-must-never-be-serialized documentation (T-2-01). | T-2-08 | Tampering | Guard.Against.Null(action) on every OnSuccess/OnFailure overload | low | mitigate | Every overload validates its action/func delegate argument via the existing Guard.Against.Null before invoking it, converting a would-be NullReferenceException into a clear ArgumentNullException at the call site — consistent with Phase 1's fail-fast discipline. Verified by this plan's null-argument test cases in both Task 1 and Task 2. |
