## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Any future caller (arbitrary predicate validation, name/label/code string fields) → Guard.Against.InvalidInput/StringTooShort/StringTooLong/InvalidFormat | The most general-purpose guards in the phase — T is unconstrained for InvalidInput, so the caller's data shape is unknown at this layer |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|------------|----------|--------------|------------------|
| T-1-01 | Tampering | GuardAgainstInputExtensions.cs / GuardAgainstStringExtensions.cs | high | mitigate | Fails fast on predicate-violating input and out-of-length/malformed strings before they reach a downstream constructor; verified by the pass/throw unit test pairs in both tasks. |
| T-1-02 | Information Disclosure | Exception messages across InvalidInput, StringTooShort, StringTooLong, InvalidFormat | high | mitigate | All messages interpolate only parameterName, never the actual rejected value — this is the highest-risk guard family for this threat since InvalidInput's T is unconstrained and could carry arbitrary (potentially sensitive) call-site data; verified by acceptance criteria requiring no interpolation of the raw value. |
| T-1-03 | Tampering (supply-chain / scope-creep) | Exclusion of a NotFound-style guard requiring a non-BCL exception | low | accept | Deliberately deferred to Phase 4 (DomainException) rather than smuggling a premature domain-exception dependency into this phase; documented inline so it is not silently reinvented ad hoc later. |

</threat_model>
