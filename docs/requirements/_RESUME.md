## Load-bearing decisions from `_DECISIONS.md` you must not contradict

- **.NET/ASP.NET Core backend** (FedRAMP/FIPS alignment) — GraphQL for internal frontend, REST for external integrations.
- **RBAC baseline + ABAC overlay** authorization model; every sensitive action gated by a **confirmation gate** (Command/Action Bus) and **step-up auth** (Authentication & Authorization) where flagged.
- **NIEM Core grounding discipline**: before inventing fields for any new entity-like concept, check the real NIEM Core type (`nc:PersonType`, `nc:ItemType`, `nc:ActivityType`, `nc:DocumentType`, `nc:EntityType`/Party, etc.) rather than guessing. This has repeatedly produced better field models than starting from scratch.
- **Entity/EntityAssociation TPT architecture** (Module 0.5): any future module that needs its own "thing with an ID that gets tracked/deduped/cross-referenced" should very likely register as an extension of one of the five base types (Party, Item, Location, Activity, Document) rather than inventing a parallel identity system. Relationships between things should very likely be a named `EntityAssociation` subtype, not a plain foreign-key field, when the relationship benefits from independent audit/history (ownership, custody, employment, etc. all do; a purely descriptive intrinsic field usually doesn't).
- **Every Entity has a display label** (`display_label_strategy`, template or computed) — a universal Entity Registry Core requirement, not per-feature.
- **Settings & Preferences** is a shared hierarchical config engine (location chain + identity chain, lockable per level) — any future tenant/site/user-configurable setting should register against it, not implement its own override logic.
- **Command system** (Command/Action Bus, Domain Events, Command Palette, CLI-Style Input) is an intentional platform differentiator — any new feature that exposes user-facing actions should register them as Command/Action Bus actions, not build bespoke buttons with no palette/CLI/automation reach.
- **"Report" = a filtered read over Activity Registry, snapshotted into an immutable Document.** Established by DAR, reused as-is by Passdown and Incident Reporting: the underlying data (DAR Entries, Incident Updates, etc.) stays live/mutable via ordinary Activities; "generating a report" is a point-in-time Document snapshot that never retroactively updates. Report *generation mode* (auto-on-approval / ad hoc-only / both) and, per Incident, report *scope mode* (one consolidated document vs. one per contributor) are both independent, tenant-configurable axes — don't assume one document per record is the only shape a future report-like feature needs.
- **A review/sign-off gate is its own governance record that never mutates what it judges** — DAR's Shift Review, Incident's Supervisor Review. Any future "did a human approve this set of records" need should follow this shape (reviewer, status, comments, scoped item list) rather than adding an approval field onto the records themselves, since those records are usually shared across other modules too.
- **Plan vs. execution: a reusable-definition hierarchy pairs one-to-one with an Activity-extension hierarchy that has a real lifecycle.** Guard Tour's Route → Tour Definition → Checkpoint (plan, versioned, no lifecycle beyond active/archived) is executed by Route Assignment → Patrol → Checkpoint Scan (Activity extensions, one per real occurrence). Plan edits are versioned; an in-progress/completed execution pins the plan version it started against — never a live FK reference, or a later edit silently rewrites what already happened. A negative outcome (a missed required occurrence) gets its own real row (Patrol's `missed` status), not a silent gap in the data.
- **Dedup/merge is for identity, not events.** Entity Registry Core's full dedup/merge/human-review workflow is for things with a real "duplicate identity" risk (a Person, an Incident). A routine timestamped event (Checkpoint Scan) doesn't need it — at most a narrow write-time debounce against an accidental double-read. Don't inherit the heavier mechanism by default onto a type it wasn't designed for.
- **`extended_fields` (Entity Registry Core) + Tenant-Defined Types & Custom Fields (Module 0)**: every `Entity`/`EntityAssociation` row carries one bounded, schema-governed JSONB escape hatch — concrete TPT columns remain the unconditional default; `extended_fields` is only for a handful of low-value optional fields or genuine tenant-defined custom fields/subtypes. **Tenant-Defined Subtypes are always anchored on a concrete developer-built type** (corrected mid-review from an earlier "registered type with no table at all" framing that skewed the user's intent): a subtype's records ARE real records of the anchor (designated via the Entity root's `tenant_subtype_ref`), inheriting the anchor's entire platform treatment — there is no free-floating tenant-defined type. Custom field keys are always `custom_`-prefixed (reserved namespace). Two non-entity carriers (Domain Events Rule, CLI Alias) already prove the mechanism generalizes beyond Master Records — any future "many small categories, each with a few optional fields" or "let a tenant self-serve a new kind of record" need should register against this, not reinvent it.
- **AI / LLM Services (Module 0)** owns all AI-provider plumbing platform-wide: SaaS-pooled tokens or tenant BYO API key (Anthropic/OpenAI/Gemini/Azure OpenAI), Prompt Templates with schema-validated `{placeholder}` substitution, and Custom Instructions resolved by a category × location lookup (a new two-axis scoping shape, distinct from Settings & Preferences' single chain). A consuming feature declares an **AI Context** (a `context_key` + its available placeholders) and never talks to a provider or assembles a prompt directly. **AI always proposes, a human always confirms** before generated content becomes part of any official record (Incident Update's `source = ai_generated` only commits on explicit approval) — this is a hard requirement for any future AI-generation feature, not a configurable default.
- **A launch-point pattern for spinning off a richer record from a lightweight one, with context pre-filled.** Checkpoint Scan → Patrol Finding/Incident, Incident escalation (`escalated_from_ref`) — any Activity that logs something in the moment should be a launch point for a fuller record via ordinary Command/Action Bus context-seeding, not a dead end.
- **Offline is a minimal capture subset, never 1:1 parity (revised decision — supersedes the hybrid CRDT + LWW design).** Offline writes follow a three-class contract: create new records (client UUID, locally editable until first sync), append rows to existing records' timelines (including sole-actor execution transitions like completing your own Patrol), and never mutate shared server-side records offline — Dispatcher on-behalf-of (radio relay) is the platform's primary disconnected workflow, elevated to a first-class cross-cutting principle. Conflicts are impossible by construction, so there is no conflict-resolution engine. When specifying any future feature, the question is "which of its Activities are append-capturable offline, if any" (default: none) — never "how does this feature work offline." Tripwire: a feature proposing offline co-editing of shared mutable state must be flagged before drafting.
- **A checked-out numbered range must fully reconcile before it's closed** (Tickets/Citations' Ticket Book: checkout → void-or-issue → reconcile, plus a `lost` terminal state) — reuse this shape for any future physical numbered-pad/tag tracking need, layered *alongside*, never instead of, a record's own digital offline-safe identity.
- **When two+ features independently need the same non-trivial capability, promote it to a shared Platform Core feature** rather than letting each build its own version — this happened four times this session (Domain Events/Command Bus/Palette split from Event & Command Bus Architecture; Tenant-Defined Types & Custom Fields promoted from a Courtesy Patrol-local field; AI/LLM Services promoted from an assumed dependency in AI-Assisted Incident Report Writing/CLI-Style Input). Treat a third instance of "I need this again" as a strong signal to check `_DECISIONS.md` before building locally.
