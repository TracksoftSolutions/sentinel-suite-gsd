# Cross-Cutting Architectural Decisions

Decisions made during feature-by-feature elicitation that apply platform-wide, beyond any single feature doc. Recorded here so later feature docs stay consistent without re-deriving them. Several of these resolve open questions from `docs/pdd.md`.

- **Backend framework: .NET.** Chosen specifically to ease FedRAMP acceptance and NIST/FIPS (crypto module) compliance for DOE/federal tenants. Resolves PDD Open Question #1 (backend language/framework). API surface pairing (REST/GraphQL conventions, versioning approach) should follow whatever is idiomatic for .NET/ASP.NET Core.
- **Application framework: loosely based on ABP (ASP.NET Boilerplate/ABP Framework), clean architecture + DDD at its core.** No longer a pending "if ABP lands" possibility (see `architecture-guidance.md` §EF Core notes and `mvp.md`'s earlier hedge) — treat ABP's layering (Domain / Application / Infrastructure / API-Host, per-module vertical slices) and its base conventions (auditing, soft-delete, tenant filtering, permission/feature/setting management, background jobs) as the starting skeleton. "Loosely based on" means: adopt the shape and the base-class/module conventions where they fit the platform's own decisions above them (TPT identity taxonomy, RBAC+ABAC, registry-authoritative capabilities); don't adopt an ABP convention that would contradict a decision already made here. Where ABP ships a module that overlaps a platform concept already specified (e.g., its Permission Management vs. this platform's RBAC+ABAC, its Setting Management vs. Settings & Preferences' location+identity chain), the platform's own spec wins — ABP's version is either adapted to match or bypassed, never left to quietly diverge as a second source of truth. Where ABP ships a module the platform hasn't specified yet (Tenant Management, Feature Management, Background Jobs, Blob Storing, Localization), that's a real elicitation gap, not an assumption that "ABP already covers it" — see Module 0 follow-up below.
- **API surface: both REST and GraphQL, different audiences.** GraphQL for the internal web/mobile frontend; versioned REST for external integrations and webhook-adjacent tooling. Both sit on the same underlying business logic/permission layer. (Established in [api-messaging-layer.md](0-platform-core/api-messaging-layer.md).)
- **Offline sync strategy: idempotent append-only outbox; offline is a minimal capture subset, never 1:1 parity.** *(Revised — originally specified as hybrid CRDT + field-level LWW; superseded during the platform design review, see the revision entry at the end of this file.)* Offline writes are limited to creating new records (client UUID, locally editable until first sync) and appending rows to existing records' timelines; shared server-side records are never mutated offline — the Dispatcher on-behalf-of (radio relay) path is the platform's primary disconnected workflow. Under single-writer + insert-only + unshared-until-sync, conflicts are impossible by construction, so no conflict-resolution engine exists. Resolves PDD Open Question #5. (See [offline-data-sync.md](0-platform-core/offline-data-sync.md).)
- **GIS/map provider: adaptor architecture, tenant-selectable.** No single hardcoded vendor; tenants pick from enabled adaptors (e.g., a self-hostable default stack, Esri, Google), with DOE/air-gapped tenants restricted to self-hostable adaptors. Resolves PDD Open Question #4. (See [gis-mapping-services.md](0-platform-core/gis-mapping-services.md).)
- **Authorization model: RBAC baseline + ABAC overlay.** RBAC grants are feature + action + data-scope. ABAC policies (subject/resource/environmental attributes) can only narrow an RBAC grant, never widen it; failed ABAC conditions hard-deny with a visible reason and an audit entry. (See [authentication-authorization.md](0-platform-core/authentication-authorization.md).)
- **MFA/authenticator model: method catalog tagged by NIST 800-63B AAL.** Passkeys, hardware keys, and CAC/PIV are strong enough to serve as standalone primary login (no separate password); TOTP app, SMS OTP, email OTP, and magic link always pair with a password as a second factor. Tenants set a minimum required AAL that gates which methods are even offered. (See [authentication-authorization.md](0-platform-core/authentication-authorization.md).)
- **Tenant isolation: one tier per Company-level tenant.** Full database isolation vs logical isolation is set once at the top of a tenant's org hierarchy (Platform Super Admin, based on client classification) and inherited by everything beneath it — not configurable per-site. (See [authentication-authorization.md](0-platform-core/authentication-authorization.md).)
- **Notifications: internal-only scope for the Platform Core Notifications Engine.** It is strictly for internal platform-user notifications (alerts, task assignments, approvals). Module 17 (Mass Notification & Crisis Communications) is a separate system for external/occupant broadcasts and may reuse channel infrastructure but not this engine's category/ack/escalation model. (See [notifications-engine.md](0-platform-core/notifications-engine.md).)
- **CQRS = Command + Query + Event, explicitly differentiated.** [event-command-bus-architecture.md](0-platform-core/event-command-bus-architecture.md) now treats the Command bus (write intent → handler → event), Query bus (read intent → read-model, RBAC/ABAC-filtered, no side effects), and Event bus (fan-out to audit/notifications/push/projections) as three distinct pillars rather than collapsing into "just an event bus."
- **Two new Platform Core features added, split out from the raw bus:** [domain-events.md](0-platform-core/domain-events.md) (business-meaningful events configured to trigger cross-module side effects) and [command-action-bus.md](0-platform-core/command-action-bus.md) (reusable, invokable action/automation catalog). Both build on top of the low-level Command/Query/Event bus infrastructure rather than being part of it.
- **Unified action registry powers every invocation surface.** The Command/Action Bus catalog is the single source of truth for "what can happen" — UI buttons, context menus, drag-and-drop, command palette, CLI-style commands, and a future true CLI all invoke the same registered actions, never separate per-surface implementations of the same effect.
- **Command Palette and CLI-Style Input specified as two separate features.** [command-palette.md](0-platform-core/command-palette.md) (fuzzy search/navigate/invoke, any user) and [cli-style-input.md](0-platform-core/cli-style-input.md) (typed `verb noun --flag value` syntax, power users). The "CLI" is explicitly an in-app command bar, not a standalone external/scriptable tool — a true external CLI remains a separate, deferred, not-yet-scoped future possibility.
- **Command/Action Bus parameters support a "default resolver."** A parameter can declare a lookup (e.g., "this site's scheduled default patrol route") that fires only when no explicit value, context, or alias/playbook template value supplies one. Final parameter resolution precedence, uniform across every invocation surface: **explicit > context > alias/playbook template > default resolver**. Added to [command-action-bus.md](0-platform-core/command-action-bus.md); consumed by [cli-style-input.md](0-platform-core/cli-style-input.md) aliases.
- **Command system "standout feature" additions (Command/Action Bus + CLI-Style Input + Command Palette).** To make the command system a genuine differentiator, several mechanisms were added, all enforced server-side and shared across every manual invocation surface:
  - **Confirmation gate**: any manual invocation of an action with no compensating action, or flagged for step-up auth, must pass an explicit confirmation (surface-appropriate UI — chip in CLI, dialog in Palette/buttons) before executing; step-up is folded into the same confirmation step. Automated (Domain Events) invocations are exempt but cannot target step-up-required actions at all.
  - **Undo history**: generalized beyond a fixed timer — an invocation with a compensating action stays undoable as long as it's the most recent invocation against its specific target entity (not global-most-recent), scoped independently per entity, with a platform backstop max age. This is what lets a Dispatcher juggling several incidents undo the last action on any one of them independently. Each surface adds its own fast/convenience presentation (e.g., a 5-second toast) on top of this shared mechanism.
  - **AI-assist mode** (CLI-Style Input only): an explicit per-entry toggle (never blended into strict parsing) translates natural language into a proposed structured command requiring confirm/edit before running; includes optional off-by-default voice input reusing the same transcription capability as AI-Assisted Incident Report Writing. Tenant-disable-able.
  - **Proactive alias suggestion**: detects a user's repeated command pattern and offers to save it as an alias.
  - **Alias sharing & promotion**: personal alias → shared to a browsable team library (opt-in adopt, no auto-behavior-change) → optionally promoted by a Site/Tenant Admin into an auto-applied site/tenant-scope default.
  - **Explicitly rejected**: physical-world (NFC/QR checkpoint scan) context-seeding for CLI-Style Input — mobile field scanning isn't a natural fit for a desktop-oriented typed-command surface. Also rejected as a headline feature: voice as a primary field-work input (minor/optional only) and a fully separate third AI input surface (folded into CLI-Style Input as a mode instead).
- **CLI-Style Input supports chained commands with implicit, type-aware context.** `;`-separated commands on one line share context automatically (e.g., `create-incident ...; assign-unit U-01 --role IncCom` assigns to the incident just created). Context is tracked per entity type (not a single most-recent-wins slot), can be ambiently seeded from the currently open record, is shown as removable chips, and explicit flags always override it. Enter executes and leaves context active; rapid double-Enter executes and clears it. A failed step halts the chain with no auto-rollback of prior steps (lighter-weight than a formally authored Command/Action Bus playbook's full saga behavior).
- **API & Messaging Layer vs internal Event/Command Bus vs Module 19 Integration Gateway.** API & Messaging Layer is the external-facing client↔server and third-party-webhook layer. The (upcoming) Event & Command Bus Architecture feature is the internal server-side CQRS mechanism those APIs publish to/read from. Module 19's Physical Security Integration Gateway owns hardware protocol adapters (BACnet, Modbus, Wiegand, RTSP) and may reuse this feature's webhook/API infra for outbound notification only. (See [api-messaging-layer.md](0-platform-core/api-messaging-layer.md).)
- **Settings & Preferences is a shared hierarchical config registry, not just a bundle of platform preferences.** Every feature registers its tenant/site/user-configurable settings against one resolution engine rather than reinventing override logic. Two independent chains: a **location chain** (Platform → Tenant → configurable sub-units → Site → configurable sub-sites → Device) and an **identity chain** (User → UserDevice). Narrowest-set-wins with upward inheritance; personal (identity chain) beats location by default for identity-anchored settings, unless an admin **locks** a setting at some level (generalizing the admin-pinned-non-mutable pattern from Notifications Engine) to block every narrower override. Device-anchored settings (e.g., a kiosk's pinned Network Profile) win over a logged-in user's personal value on shared hardware; identity-anchored ones (e.g., language) don't. Setting changes are versioned and audit-tier. Retroactively, earlier docs' ad hoc policy models (Authentication's Tenant Auth Policy, GIS's GPS Retention Policy, Notifications' Tenant/User Notification Policy) are Setting Definitions registered against this engine, not standalone mechanisms. See [settings-preferences.md](0-platform-core/settings-preferences.md).
- **Entity modeling aligns with NIEM Core as an internal discipline, not full conformance (yet).** Entity/extension type naming and structure follow NIEM Core's taxonomy (`nc:Person`, `nc:Location`, `nc:Item`, `nc:Organization`) so a future NIEM-conformant exchange is a mapping exercise, not a rework — motivated by a named national-lab (DOE) client and plausible future CAD-to-CAD mutual aid interoperability with external fire/police departments (many CAD systems use NIEM-based standards) and CJIS-adjacent law enforcement BOLO sharing. Full IEPD/XML schema conformance is explicitly deferred until a concrete external exchange partner is identified. (See [entity-registry-core.md](0.5-master-records/entity-registry-core.md).)
- **Entity Registry Core is generic/extensible, base-entity-plus-extension pattern.** Any feature can register an Entity Type (Person, Vehicle, Location today) or an Extension Type (e.g., Employee/Visitor/Contractor/BOLO-Subject/Incident-Participant extending Person) against a shared core registry — mirroring the registration pattern already used for Settings and Command/Action Bus. Extensions add both fields and their own independent lifecycle; one base entity can carry multiple simultaneous extensions. Entity IDs are strictly tenant-scoped (never shared across the isolation boundary); a subcontractor's cross-tenant identity is linked via Authentication's existing multi-tenant identity bridge, not by merging Entity Registry data. Deduplication is always human-reviewed — no auto-merge at any confidence level, given the liability risk of misattributing history (e.g., BOLO status) to the wrong person. Merges redirect references to a canonical survivor, tombstone the loser, and are reversible with full audit trail.
- **Entity Registry Core is a service layer, not a generic data table — and Activity is a fourth base Entity Type.** Retrofitted after the user pushed on "why not just Person/Item/Conveyance/Location/Activity as concrete bases": Entity Registry Core now explicitly provides shared *behavior* (ID issuance, dedup, merge/tombstone, BOLO flag, tenant-scoping) via a lightweight cross-reference record (`Entity`: entity_id + type + status + a pointer to the real record), while Person/Item/Location each own their own concrete, strongly-typed schema with the actual fields — avoiding the entity-attribute-value anti-pattern a literal generic table would create. **Activity** (`nc:ActivityType` — calls, incidents, citations, accidents, alarms, inspections, dispatches, drills) joins Person/Item/Location as a fourth base Entity Type, established now (thin registration pattern only) specifically because it's what makes Entity Relationships & History's cross-module timeline a single uniform join instead of a patchwork of ad hoc foreign keys — and because Activity subtypes are, per the user, "the whole point of the platform." Individual modules (Dispatch/CAD, Security Operations, Physical Security Integration Gateway, Safety Management, etc.) will register their own Activity extensions with rich fields when those modules are specified. (See [entity-registry-core.md](0.5-master-records/entity-registry-core.md).)
- **Master Records base type set rounded out to five: Party, Item, Location, Activity, Document.** Prompted by the user pointing out NIEM Core has real, distinct `nc:EntityType` (Person-or-Organization), `nc:OrganizationType`, and `nc:DocumentType` bases that were missing. Key structural changes:
  - **Party** (renamed from NIEM's "Entity" to avoid colliding with "Entity Registry Core," the platform's own service-layer name — "Party" is standard enterprise-data-modeling terminology for exactly this Person-or-Organization duality) becomes a genuine base Entity Type with its own identity/dedup/merge, registered directly against Entity Registry Core. **Person is retrofitted to extend Party** rather than being a directly-registered base type — a real architectural change, not just a rename: Person Registry no longer owns `contact_information`/`identifications[]` (inherited from Party), and extension chains can now nest more than one level (Employee → Person → Party).
  - **Organization** (new) is Party's second extension, uniform/no-category like Vehicle, with basic `parent_organization_ref` self-nesting. A tenant can designate one Organization record as itself, letting fields like Vehicle's owner resolve to a real Party reference instead of a three-way person/tenant-fleet/external-org branch — Vehicle/Conveyance Registry's `owner_type` enum was retired in favor of `owner_party_ref` (nullable, defaults to tenant-owned).
  - **Document** (new) is the fifth base type, `nc:DocumentType`-aligned, with hash/integrity verification and version history built at the base level — directly serves Investigation Management's future "Digital Evidence Locker" (already named in MODULES.md) rather than that module reimplementing file-integrity tracking.
  - **Entity Relationships & History synthesizes the whole Master Records architecture into two views**: an **Interaction Timeline** (chronological — Activity participation/attachment/location, EntityAssociation transitions, BOLO Flag lifecycle) and a **Relationship Snapshot** (current-state — only active EntityAssociation rows, grouped by kind). Both are CQRS read-model projections (not live queries), require a dedicated Interaction Timeline Viewer permission layered on top of (never bypassing) each entry's own ABAC/sensitivity filtering, and need no special-case merge handling since Entity Registry Core's merge already redirects historical *and* active association rows to the survivor. Scheduled shifts and access-control events (named in MODULES.md) are an explicitly deferred integration point, not built now, since Personnel and Access Control aren't specified yet. This is the last Master Records feature. (See [entity-relationships-history.md](0.5-master-records/entity-relationships-history.md).)
- **Final architecture: Table-Per-Type (TPT) inheritance for both Entity and EntityAssociation, one shared PK per hierarchy.** Third and final refinement of the entity/association architecture, prompted by the user proposing TPT directly. Two independent TPT hierarchies: **Entity** (`Entity` → `Party` → `{Person, Organization}` → `{Employee, Visitor, ...}`, and `Entity` → `{Item, Location, Activity, Document}` → their own extensions) and **EntityAssociation** (`EntityAssociation` → named concrete kinds like `ConveyanceOwnerAssociation`, `CustodyAssociation`). At every level, the row's primary key **is** `entity_id` (or `association_id`) — shared with and FK'd to the parent level — eliminating the earlier redundant `extension_id` + `entity_id` pointer pair, and giving associations real named types instead of a string `association_type` discriminator alone (the discriminator is kept only as a practical query aid, not the source of truth for type). This also resolved the user's sharp challenge to the earlier "generic vs. concrete" framing: the `Entity`/`EntityAssociation` root tables ARE generic tables, and that's fine specifically because they hold only uniform metadata (id/type/status, or two-refs/role/audit) — genericity earns its keep exactly where heterogeneity is the modeled feature (identity, arbitrary relatability) and costs you exactly where it isn't (the actual field data, which stays in concrete per-level tables).
  - **Full sweep**: every direct entity-reference field across Master Records was re-evaluated and converted to a named EntityAssociation subtype where the relationship benefits from independent history/audit: `ConveyanceOwnerAssociation` (Vehicle ownership, replacing `owner_party_ref`), `CustodyAssociation` (Item custody, replacing `current_holder_ref`+`custody_history[]` — current holder is just the active row, history is the active/removed sequence), `EmergencyContactAssociation` (Person-to-Person, replacing the dual-mode embedded field — the contact must now be a real, possibly minimal, Person entity, no free-text fallback), `HierarchyAssociation` (one reusable subtype for both Location's and Organization's parent/child nesting, replacing two separate `parent_*_ref` fields), `ActivityLocationAssociation` (replacing Activity's `location_ref`, gaining location-correction history for free), `DocumentAuthorAssociation` (replacing `author_party_ref`, and now supports multiple co-authors for free), `FacilityManagerAssociation` (replacing Location's `facility_manager_contact_ref`), `ActivityParticipantAssociation`/`ActivityAttachmentAssociation` (named subtypes replacing the earlier discriminator-only rows).
  - **Display labels are universal, owned by Entity Registry Core** (corrected after an initial misread): every registered Entity Type/Extension Type, at every TPT level — Party, Person, Organization, Item, Vehicle, Location, Activity and every Activity extension, Document — must declare a `display_label_strategy`, either `template` (a simple field readout, e.g. Person → full name, Vehicle → plate+make/model, Location → location_name, Document → document_title) or `computed` (synthesized summary logic, needed by Activity extensions specifically, since a useful label there draws on several fields, not one). The requirement explicitly does **not** extend to non-entity system/governance records (`EntityAssociation` rows, `PotentialDuplicate`, `MergeRecord`, `BOLOFlag`, audit-log entries) — those are metadata *about* entities, not entities, and render using the labels of whatever entities they reference. Consumed by Command Palette, CLI-Style Input context chips, and Entity Relationships & History's timeline uniformly, with no per-surface type-switching logic.
  (See the retrofitted [entity-registry-core.md](0.5-master-records/entity-registry-core.md) and every other Master Records doc, all updated to match.)
- **Associations: one real base table anchored at the Entity layer, refined twice.** First pass specialized associations into bespoke concrete tables per pairing (`ActivityPartyAssociation`, etc.) to avoid a polymorphic-FK problem. User pointed out a better resolution: anchor both sides of a relationship (`entity_id_a`, `entity_id_b`) to Entity Registry Core's own lightweight `Entity` cross-reference record — not to the N concrete schemas directly — which makes "points to a valid Entity" a real, single-table, enforceable constraint. This yields one genuinely generic, physical **Entity Association** table (`entity_id_a`, `entity_id_b`, `association_type`, `role`, `added_by`, `added_at`) that isn't the EAV anti-pattern, because unlike a field-blob table, an edge-list (two refs + a role) is uniform data regardless of what it connects — a standard graph/join-table pattern. Specialization now means exactly what it means for entities: an **Entity Association Extension** (its own small table, FK'd to the base association row) only when a specific `association_type` needs more than role+audit (e.g., `employment` needing start/end dates) — mirroring Employee-extends-Person-extends-Party. Activity Registry's three bespoke tables were retired in favor of Entity Association rows with `association_type` = `activity_participant` / `activity_attachment`. Confirmed NIEM-faithful throughout: `nc:AssociationType` is itself a generic role+entity-reference pattern used across NIEM, not something Person-specific.
  (See [party-registry.md](0.5-master-records/party-registry.md), [organization-registry.md](0.5-master-records/organization-registry.md), [document-registry.md](0.5-master-records/document-registry.md), retrofitted [entity-registry-core.md](0.5-master-records/entity-registry-core.md), [person-registry.md](0.5-master-records/person-registry.md), [activity-registry.md](0.5-master-records/activity-registry.md), [vehicle-conveyance-registry.md](0.5-master-records/vehicle-conveyance-registry.md).)
- **Activity Registry promoted to its own full first-class doc**, giving the fourth base Entity Type (established thin in Entity Registry Core) the same depth as Person/Item/Location. Owns three generalized mechanisms: (1) a **generic Participant association** (any Person or Item entity, tagged with a role, on any Activity) — confirmed NIEM-faithful via `nc:AssociationType`'s generic role+entity-reference pattern, used throughout NIEM for exactly this purpose; this **retires Person Registry's separate "Incident Participant" extension**, which is removed in favor of the shared mechanism (same generalization instinct as BOLO Flag). (2) **Offline-safe identification** reusing Offline Data Sync's existing client-UUID + server-assigned-sequential-number pattern unmodified. (3) **Dedup/merge** using the same tombstone + redirect + reversible model as Person/Item/Location, explicitly designed to power Dispatch/CAD's future "Incident Merging" feature (already named in MODULES.md) rather than that module reimplementing it. This doc establishes only the base mechanics; individual modules (Dispatch/CAD, Security Operations, Physical Security Integration Gateway, Safety Management, Emergency Management) register their own rich Activity extensions (Call, Incident, Citation, Accident, Alarm, Inspection, Drill) when specified. (See [activity-registry.md](0.5-master-records/activity-registry.md).)
- **Location Registry grounded in `nc:LocationType`, with a base-level `location_type` field** (unlike Vehicle's deliberate no-category pattern) since geometry needs genuinely differ by kind (address for a Site, polygon for a Zone, floor/room identifiers for a Room) — structural, not purpose-driven. Carries basic `parent_location_ref` self-nesting as an intrinsic structural fact (Room→Floor→Building), with the rich hierarchy-building UI deferred to Facility & Zone Management's future Location Hierarchy Designer. "Utility & Contact Directory" gets a light quick-reference surface here (facility manager contact, shutoff pointers, local dispatch number); the full directory is deferred to Facility & Zone Management's future Utility Control Tracking — same light-reference/deferred-lifecycle pattern as Vehicle's `associated_permits[]`. **Location geometry vs. GIS Geofence** are kept as distinct concepts (a location's footprint = intrinsic identity; a geofence = a purpose-built alerting overlay that doesn't need 1:1 correspondence) but a Geofence may now optionally set `derived_from_location_ref` to reuse a location's own geometry instead of requiring a redundant independently-drawn duplicate shape (touched up in [gis-mapping-services.md](0-platform-core/gis-mapping-services.md)). (See [location-registry.md](0.5-master-records/location-registry.md).)
- **Vehicle/Conveyance Registry promoted to its own full first-class doc**, sitting on top of Item Registry rather than being a thin stub within it — mirroring Person Registry's depth, since Vehicle (like Person) is a cross-cutting core concern consumed by many modules (Dispatch/CAD, Access Control, Executive Protection, Supply Chain, Special Event) with no single owner. Base Vehicle record is deliberately **uniform, no category enum** — a vehicle's purpose (fleet/personal/visitor/vehicle-of-interest) is expressed by which further extensions consuming modules attach later, exactly like Person's Employee/Visitor/Contractor pattern, not a flat field. Introduces **owner** (who the vehicle is registered to — person, tenant fleet, or a future Organization type) as explicitly distinct from Item Registry's existing **custody** (who currently has/is operating it) — a fleet car is owned by the tenant but custody rotates shift to shift. Permit/parking-pass/authorization-tag association is a lightweight reference here (`associated_permits[]`); full lifecycle deferred to Access Control. (See [vehicle-conveyance-registry.md](0.5-master-records/vehicle-conveyance-registry.md).)
- **"Vehicle Registry" generalized into "Item Registry" (base `nc:ItemType` + Vehicle as its first `nc:ConveyanceType`-aligned extension).** Same two-layer pattern as Person Registry (not a third infrastructure layer) — Item Registry defines the base Item entity directly against Entity Registry Core. Item carries generic **custody/possession tracking** (current holder + transfer history, atomic and audit-tier) at the base level, anticipating that Weapons Inventory, Key Custody, Lost & Found, Evidence Tracking, and K9 Equipment will all register their own Item extensions later and inherit this rather than rebuilding chain-of-custody logic each time. Vehicle's own fields (plate, jurisdiction, model year, body style) are additive to the base Item's make/model/color/photo, never duplicating them. (See [item-registry.md](0.5-master-records/item-registry.md).)
- **BOLO Flag generalized to Entity Registry Core, applicable to any entity type.** Originally built Person-specific in Person Registry, then retrofitted: one governed mechanism (Supervisor+ permission, mandatory justification, mandatory expiration, step-up auth to create/clear, audit-tier) lives at Entity Registry Core and is reused unmodified by Person's BOLO/Trespass Subject and Vehicle's BOLO/Violation Flag — consuming entity types add only their own context field (e.g., trespass_notice_ref, citation_ref). Same generalization instinct as the Command/Action Bus confirmation gate and undo history (put shared governance at the lowest common layer, not duplicated per consumer). (See [entity-registry-core.md](0.5-master-records/entity-registry-core.md).)
- **Person Registry's base fields are structurally grounded in `nc:PersonType`, not an invented field list.** Structured name with aliases, NIEM-style physical description (sex, race/ethnicity, height/weight/eye/hair color, distinguishing features), a generic repeatable Identification structure (id value + category, covering SSN/DL/passport/badge/etc.), associated Contact Information, and Person-to-Person emergency-contact associations. This is a materially better fit for BOLO/witness-description use cases than a generic HR-style profile. Medical alerts and language fluencies have no NIEM Core equivalent and are explicitly labeled as platform-specific augmentations. Race/ethnicity and citizenship are on-by-default nullable fields (tenant-disableable, sensitivity-classified) per explicit user direction: "this platform is about security not political correctness," matching real BOLO/witness-description practice and NIEM Core itself. These live once on the base Person entity (never duplicated per extension — they're intrinsic characteristics of the person, not the relationship) with **field presence decoupled from workflow display**: each extension type's owning module decides whether its own create/edit UI surfaces these fields by default (e.g., Personnel's Employee flow need not prompt for race; Access Control's BOLO flow should). This is the primary control against US-domestic employment-law exposure (EEOC disparate-treatment risk, state sensitive-PI statutes) without weakening the data model or the BOLO use case; GDPR was considered and correctly ruled not operative given the platform's US-commercial/DOE-national-lab target market. Exact NIEM version/element-name verification remains a technical-spec task. This same grounding approach (checking the real NIEM Core type before inventing fields) should be applied when specifying Vehicle Registry (`nc:Item`/vehicle extension) and Location Registry (`nc:Location`). (See [person-registry.md](0.5-master-records/person-registry.md).)
- **Terminology: "Tenant" is reserved exclusively for the SaaS multi-tenancy concept.** MODULES.md's Person Registry profile type "tenants" (building occupants/residents) is renamed **Occupant** platform-wide to avoid collision — consumed primarily by Module 9's Tenant & Occupant Registry feature (which itself should probably be read/renamed with this in mind when specified). (See [person-registry.md](0.5-master-records/person-registry.md).)
- **Person Registry extension types get only a core/shared skeleton here; richer fields deferred to owning modules.** Employee, Visitor, Contractor, and Occupant extensions each have minimal identifying fields defined in Person Registry; Personnel, Access Control, Subcontractor Management, and Facility & Zone Management respectively add their own richer fields when those modules are specified — consistent with Entity Registry Core's extensible registration pattern.
- **BOLO/Trespass Subject is a governed, high-liability extension type**: elevated permission (Supervisor+) required to create/activate, mandatory justification, mandatory expiration (statutory or tenant-configured default), and step-up authentication required for both creation and clearance.
- **Sensitive Person fields (medical alerts, emergency contacts) use emergency-aware visibility**: restricted to a specific permission under normal operations, automatically broadened to EOC/muster-coordinating roles during an active emergency event, then reverted — a pattern other sensitive-field features may want to reuse.
- **Network Profiles are a Settings & Preferences concept**: bundled bandwidth behaviors (media sync deferral, compression, tile pre-cache, sync frequency, push verbosity), platform presets + tenant-customizable, auto-selected by connection quality with manual pin override at any eligible hierarchy level.
- **First Module 1 (Security Operations) feature: Daily Activity Reports (DAR).** Established three reusable patterns for any future "report" feature (SITREPs, After-Action Reports, and others are likely candidates):

- **Big-picture review of Guard Tour surfaced five real gaps, all resolved by the user, worth carrying forward as general lessons:**
  - **A checkpoint doesn't belong to any one tour.** A Checkpoint can be referenced by multiple Tour Definitions (even across Routes) — so anything that's a property of the physical checkpoint (required verification method, inspection instructions) belongs on the Checkpoint, never on whichever plan level currently references it. General lesson: when a "child" record can legitimately have multiple "parents," don't let parent-owned config leak onto the child from just one of them.
  - **Starting an execution-level occurrence needs to be an explicit, trackable moment** ("the system needs to know when Unit 1 starts a patrol of building 37"), not inferred — resolved as explicit start (self, or on-behalf-of via radio, reusing Patrol Management's existing radio-check posture) OR implicit start via first scan **only when unambiguous**; an ambiguous match (a checkpoint shared by multiple currently-eligible tours) prompts the user rather than guessing. General lesson: never let the system silently disambiguate against a real-world decision a human actually needs to make.
  - **Plan mutability during active execution needs versioning, not live references.** Route/Tour Definition edits create a new version (same discipline as Settings & Preferences); Route Assignment/Patrol pin the version in effect when they started, so an in-progress or completed occurrence's requirements never shift retroactively under a later plan edit. General lesson: any plan/execution split needs an explicit answer to "what happens to in-flight executions when the plan changes" — a live FK reference is usually the wrong default.
  - **Verification/compliance checks should degrade gracefully, never block completion.** A guard can always manually override a failed/unavailable verification method, recorded as unverified rather than blocking the work. General lesson: a compliance-tracking mechanism must not become an operational blocker — track the gap, don't prevent the job.
  - **Dedup is for identity, not events.** Checkpoint Scan doesn't need Entity Registry Core's full dedup/merge/human-review machinery — only a narrow write-time debounce against an accidental immediate double-read. Flagged for Activity Registry to eventually formalize a "high-volume event-type Activities may opt out of dedup" allowance. General lesson: don't inherit a shared mechanism's full weight by default onto a type it wasn't designed for — a routine timestamped event has no "duplicate identity" the way a Person or Incident does.
  - **A negative/failure outcome should still be a real, queryable record, not a silent absence.** A missed required occurrence auto-creates a `missed` Patrol row (with an optional context-only resolution note) rather than just leaving a gap in the data — "missed points are missed points," no waiver state that softens the record. - **Offline architecture revised during platform design review: "1:1 offline parity" rejected as a requirement; hybrid CRDT + field-level LWW retired.** Per explicit user direction, grounded in 20+ years of security-management domain experience: most facilities — including DOE underground areas — have working WiFi; the truly offline zones (SCIFs and similar) are places a device typically cannot enter at all, which is exactly why the platform's Dispatcher on-behalf-of capability was designed to cover ~95% of an officer's system actions via radio relay. Offline device capability is therefore a **minimal capture subset with accepted degraded performance**, governed by a three-class contract: (1) **create new records offline** (client UUID, freely editable locally until first sync — single-writer and unshared, so safe); (2) **append rows to existing records' timelines** (notes/updates/acknowledgments, plus state-transition events only on execution records the user is the sole assigned actor of, e.g., their own Patrol); (3) **never mutate shared server-side records offline** (Incident status/severity/assignment, plan definitions, settings — online or Dispatcher-relayed only; an offline officer can at most *append* a "requested status change" timeline event for later confirmation). Key consequences and lessons:
  - **Conflicts are impossible by construction** under single-writer + insert-only + unshared-until-sync, so the entire conflict-resolution apparatus (field-level LWW engine, supervisor reconciliation queue, conflict flags) is **deleted as a requirement, not deferred**. What remains is boring and verifiable: a durable outbox, per-device monotonic sequence numbers (no vector clocks — single writer per device), dual timestamps (client-claimed + server-received, drifting field clocks assumed), idempotent replay keyed by client UUID, and batch-internal UUID reference resolution.
  - **Multi-officer concurrent notes on one Incident are not a conflict case**: the update-timeline architecture (each Incident Update its own immutable row) already converted would-be shared mutation into independent appends — set-union merge, interleaved for display by client-claimed time with server-received tiebreak. This is a quiet payoff of the "record = timeline of updates" report architecture.
  - **Stale-context authorship is accepted, not fixed**: an offline officer appending a note without having seen concurrent entries is radio-era operational reality; the record honestly shows what each author knew and when. No merge complexity exists to paper over it.
  - **Air-gapped facilities are a deployment-model concern, not a client-sync concern** — they run a self-hosted local instance and clients there are online to it. The prior doc's multi-day fixed-terminal offline requirement is retired accordingly.
  - **Tripwire**: any future feature proposing offline *co-editing of shared mutable state* reopens this decision and must be flagged against this file before drafting; the default answer is "radio it in."
  - **The Dispatcher on-behalf-of pattern is elevated to a first-class cross-cutting design principle**, not a per-feature convenience: future modules must preserve a connected actor's ability to perform a disconnected officer's workflow actions on their behalf, because it — not device-side sync — is the platform's primary disconnected path.

- **Merge/unmerge state machine fully specified in Entity Registry Core, closing the design review's largest open defect.** The prior spec was three sentences ("redirect every reference," "transactionally safe," "reversible") hiding a distributed-consistency problem and an unanswered post-merge-reference question. Resolved through elicitation (four user decisions) plus engineering drafting:
  - **Instant switchover + background rewrite** (user choice): confirming a merge executes one atomic transaction (tombstone + `merged_into_ref` + survivorship writes); from that instant the merge is behaviorally complete — reads resolve redirects transitively, writes arriving with a tombstoned reference are re-pointed at write time, and an idempotent, resumable background job physically rewrites stored references over the following minutes. "Transactionally safe" is achieved by construction (behavioral invisibility of the rewrite), not by one giant lock — the NFR was rewritten to say so honestly.
  - **Field survivorship: field-by-field pick with survivor default** (user choice, hybrid of the offered options): the confirmation flow presents only genuinely conflicting fields (all shared TPT levels + colliding `extended_fields` keys), survivor's value pre-selected, admin flips individual fields; every choice logged in `survivorship_log[]`, which is also what makes reversal's field restoration possible. Same-extension-type collisions resolve the same way; loser-only extensions carry over wholesale.
  - **BOLO Flags always carry over unmodified, both sides'** (user choice): each remains an independent governed record; a redundant pair triggers a post-merge review notification, and clearance goes through the standard Supervisor+/step-up gate — flag governance is never delegated to the merge flow or the merging admin.
  - **Reversal: restored Entity goes active immediately; post-merge references are admin-allocated, never defaulted** (user choice): references created during the merged period land on an allocation worklist and the reversal only reaches `reversal_completed` when every item is explicitly assigned to survivor or restored Entity — misattribution is precisely the harm reversal exists to fix. Interim state is honest and visible: both records operable, unallocated references still resolving to the survivor, flagged on both records. Chained merges reverse LIFO.
  - **Reference Field Registration** (new build-time catalog): every direct entity-reference field across every module registers itself, mirroring the platform's established registration discipline — this is what turns "redirect every reference across every module" from an aspiration into a mechanically enumerable operation, resolving the review's "implies a global registry of referencing columns that no doc specifies" gap. EntityAssociation rows need no registration (structurally enumerable).
  - **Post-redirect association collisions are auto-resolved and logged, surfaced in a post-merge review summary**: exact duplicates collapse (keep earliest); single-current-value kinds keep most-recent-active and soft-remove the older (the existing active/removed history mechanism absorbing the collision); a row made self-referential by the redirect is soft-removed and flagged — a useful late signal the "duplicates" may actually have been two related people.
  - General lesson: a mechanism described as "transactionally safe" across module boundaries in a CQRS platform almost always actually means "atomic switchover + convergent background repair + read-time indirection" — spec that shape explicitly rather than leaving "transactional" to be discovered as impossible during implementation.
